Diagrama de Clases UML

El Diagrama de Clases es el diagrama más utilizado en UML. Representa la estructura estática del sistema mostrando las clases, sus atributos, métodos y las relaciones entre ellas.

Componentes Principales:

  • Clase: Plantilla para crear objetos
  • Atributos: Propiedades de la clase
  • Métodos: Comportamientos de la clase
  • Relaciones: Asociación, Herencia, Composición, Agregación
Notación
Importante: La visibilidad se indica con símbolos: + público, - privado, # protegido, ~ paquete.

Ejemplo Práctico: Sistema de Biblioteca

Diagrama de Clases

Interpretación:
  • Un Usuario puede realizar uno o más Préstamos
  • Cada Préstamo incluye uno o más Libros
  • Los atributos privados (-) solo son accesibles dentro de la clase
  • Los métodos públicos (+) pueden ser llamados desde otras clases

Diagramas de Paquetes

Los Diagramas de Paquetes organizan elementos del modelo en grupos relacionados, mostrando la arquitectura de alto nivel del sistema.

Características:

  • Agrupan clases relacionadas
  • Definen la estructura modular
  • Muestran dependencias entre paquetes
  • Facilitan la organización del código
  • Mejoran la mantenibilidad
Ventaja: Permiten visualizar la arquitectura del sistema de forma clara y organizada.
Notación de Paquete

Ejemplo: Sistema E-Commerce

Estructura de Paquetes

Las flechas discontinuas (- - ->) indican dependencias entre paquetes. La arquitectura en capas facilita el mantenimiento y la escalabilidad.

Modelo Entidad-Relación (MER)

El MER es un modelo conceptual para diseñar bases de datos. Representa entidades, atributos y relaciones entre datos.

Componentes del MER:

1 Entidades

Objetos o conceptos del mundo real que tienen existencia independiente. Se representan con rectángulos.

Ejemplo: Cliente, Producto, Pedido

2 Atributos

Propiedades o características de las entidades. Se representan con óvalos.

Tipos: Simples, Compuestos, Derivados, Multivaluados

3 Relaciones

Asociaciones entre entidades. Se representan con rombos.

Cardinalidad: 1:1, 1:N, N:M

Símbolos MER

Ejemplo MER: Sistema de Ventas

Diagrama Entidad-Relación

Interpretación:
  • Un CLIENTE puede realizar muchos (N) PEDIDOS → Relación 1:N
  • Un PEDIDO pertenece a un (1) CLIENTE
  • Un PEDIDO puede contener muchos (N) PRODUCTOS → Relación 1:N
  • Los atributos subrayados representan las claves primarias

Modelo de Datos Relacional (MDR)

El MDR transforma el MER en un esquema de tablas relacionales implementable en una base de datos.

Conceptos Clave:

Tabla (Relación)

Estructura bidimensional compuesta por filas y columnas.

Clave Primaria (PK)

Atributo o conjunto de atributos que identifican únicamente cada fila.

Clave Foránea (FK)

Atributo que referencia la clave primaria de otra tabla.

Reglas de Transformación:

  • Cada entidad → una tabla
  • Atributos simples → columnas
  • Relación 1:N → FK en lado "N"
  • Relación N:M → tabla intermedia
  • Atributos multivaluados → tabla separada
Consejo: Normalizar las tablas para eliminar redundancias y mantener integridad de datos.

Ejemplo MDR: Sistema de Ventas

Diccionario de Clases

El Diccionario de Clases es una documentación detallada que describe cada clase del sistema, sus responsabilidades, atributos y métodos.

¿Qué incluye?

  • Nombre de la clase: Identificador único
  • Descripción: Propósito y responsabilidad
  • Atributos: Nombre, tipo, visibilidad, descripción
  • Métodos: Firma, parámetros, retorno, descripción
  • Relaciones: Asociaciones con otras clases
  • Restricciones: Reglas de negocio

Beneficios:

Comunicación clara

Facilita el entendimiento entre desarrolladores

Mantenibilidad

Simplifica modificaciones futuras

Onboarding

Ayuda a nuevos miembros del equipo

El diccionario de clases debe mantenerse actualizado durante todo el ciclo de vida del proyecto.

Diccionario de Datos

El Diccionario de Datos documenta la estructura de la base de datos: tablas, columnas, tipos de datos, restricciones y relaciones.

Información que contiene:

Tablas
  • Nombre de la tabla
  • Descripción y propósito
  • Lista de columnas
Columnas
  • Nombre, tipo de dato, longitud
  • Nulabilidad (NULL / NOT NULL)
  • Valores por defecto
  • Restricciones (CHECK, UNIQUE)
Claves y Relaciones
  • Claves primarias (PK)
  • Claves foráneas (FK)
  • Índices
¿Por qué es importante?
  • Documentación centralizada
  • Facilita el desarrollo
  • Estandarización de nomenclatura
  • Base para migraciones
  • Auditoría y cumplimiento
Importante: Debe actualizarse con cada cambio en el esquema de la base de datos.

Mejores Prácticas

Para Diagramas:

1 Claridad y Simplicidad

Evita diagramas sobrecargados. Si es complejo, divídelo en varios diagramas.

2 Nomenclatura Consistente

Usa convenciones de nombres estándar (CamelCase para clases, snake_case para BD).

3 Documentación Actualizada

Mantén los diagramas sincronizados con el código y la base de datos.

4 Usa Herramientas Apropiadas

Enterprise Architect, Draw.io, Lucidchart, PlantUML, dbdiagram.io

Para Diccionarios:

1 Descripciones Claras

Cada elemento debe tener una descripción que cualquiera pueda entender.

2 Completitud

No omitas información. Incluye todos los atributos, restricciones y relaciones.

3 Versionado

Mantén un historial de cambios con fechas y responsables.

4 Accesibilidad

Guarda en repositorio compartido (Git, Confluence, Wiki interna).

"El mejor código es aquel que está bien documentado. Los diagramas y diccionarios son parte fundamental de esa documentación."

Código SQL de Implementación

Ejemplo de creación de tablas basado en el MDR:

-- Crear tabla USUARIO
CREATE TABLE USUARIO (
    id_usuario INT PRIMARY KEY AUTO_INCREMENT,
    nombre VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,
    fecha_registro DATETIME DEFAULT CURRENT_TIMESTAMP,
    activo BOOLEAN DEFAULT TRUE
);

-- Crear tabla CATEGORIA
CREATE TABLE CATEGORIA (
    id_categoria INT PRIMARY KEY AUTO_INCREMENT,
    nombre VARCHAR(50) NOT NULL UNIQUE,
    descripcion TEXT
);

-- Crear tabla LIBRO
CREATE TABLE LIBRO (
    id_libro INT PRIMARY KEY AUTO_INCREMENT,
    isbn VARCHAR(20) UNIQUE NOT NULL,
    titulo VARCHAR(200) NOT NULL,
    autor VARCHAR(100) NOT NULL,
    editorial VARCHAR(100),
    anio_publicacion INT,
    tipo ENUM('fisico', 'digital') NOT NULL,
    -- Para libros físicos:
    ubicacion VARCHAR(50),
    estado VARCHAR(20),
    -- Para libros digitales:
    formato VARCHAR(10),
    tamanio_mb DECIMAL(10,2),
    url_descarga VARCHAR(255),
    disponible BOOLEAN DEFAULT TRUE,
    CONSTRAINT chk_anio CHECK (anio_publicacion > 1000)
);

-- Crear tabla PRESTAMO
CREATE TABLE PRESTAMO (
    id_prestamo INT PRIMARY KEY AUTO_INCREMENT,
    id_usuario INT NOT NULL,
    id_libro INT NOT NULL,
    fecha_prestamo DATETIME DEFAULT CURRENT_TIMESTAMP,
    fecha_devolucion_esperada DATE NOT NULL,
    fecha_devolucion_real DATETIME,
    estado ENUM('activo', 'devuelto', 'vencido') DEFAULT 'activo',
    FOREIGN KEY (id_usuario) REFERENCES USUARIO(id_usuario),
    FOREIGN KEY (id_libro) REFERENCES LIBRO(id_libro)
);

-- Crear tabla MULTA
CREATE TABLE MULTA (
    id_multa INT PRIMARY KEY AUTO_INCREMENT,
    id_prestamo INT NOT NULL,
    monto DECIMAL(10,2) NOT NULL,
    fecha_generacion DATETIME DEFAULT CURRENT_TIMESTAMP,
    pagada BOOLEAN DEFAULT FALSE,
    FOREIGN KEY (id_prestamo) REFERENCES PRESTAMO(id_prestamo)
);

-- Tabla intermedia para relación N:M
CREATE TABLE LIBRO_CATEGORIA (
    id_libro INT,
    id_categoria INT,
    PRIMARY KEY (id_libro, id_categoria),
    FOREIGN KEY (id_libro) REFERENCES LIBRO(id_libro),
    FOREIGN KEY (id_categoria) REFERENCES CATEGORIA(id_categoria)
);

-- Crear índices para mejorar rendimiento
CREATE INDEX idx_usuario_email ON USUARIO(email);
CREATE INDEX idx_libro_titulo ON LIBRO(titulo);
CREATE INDEX idx_prestamo_usuario ON PRESTAMO(id_usuario);
CREATE INDEX idx_prestamo_estado ON PRESTAMO(estado);
Listo! Este esquema implementa todas las relaciones y restricciones definidas en el modelo.
