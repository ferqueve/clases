Ciclo de Vida del Software

Definición

El Ciclo de Vida del Software (SDLC - Software Development Life Cycle) es un proceso estructurado que describe las etapas por las que pasa un sistema de software desde su concepción inicial hasta su retiro. Proporciona un marco de trabajo para planificar, crear, probar, implementar y mantener software de alta calidad.

Objetivos del SDLC

  • Calidad: Producir software confiable y eficiente
  • Tiempo: Entregar dentro del cronograma establecido
  • Costo: Mantenerse dentro del presupuesto
  • Satisfacción: Cumplir expectativas del cliente

Modelos Populares

  • Cascada: Secuencial y lineal
  • Incremental: Entregas por partes
  • Espiral: Iterativo con análisis de riesgos
  • Ágil: Iterativo y adaptativo

Etapa de Análisis

¿Qué es la Etapa de Análisis?

La etapa de análisis es la segunda fase del ciclo de vida del software, donde se estudian en profundidad las necesidades del negocio y se definen los requisitos funcionales y no funcionales del sistema. Es el puente entre "¿qué queremos?" (planificación) y "¿cómo lo construimos?" (diseño).

Frase Clave

"El análisis no es adivinar lo que el cliente quiere, es descubrir lo que el cliente necesita, aunque él mismo no lo sepa expresar."

Resultado Esperado

Al finalizar el análisis, debe existir un documento de especificación de requisitos (SRS) que sirva como contrato entre el equipo de desarrollo y el cliente. Este documento debe ser:

✓ Completo: Cubre todos los requisitos

✓ Consistente: Sin contradicciones

✓ Verificable: Se puede probar

✓ Modificable: Fácil de actualizar

✓ Trazable: Cada requisito tiene origen identificado

¿Cuál Técnica Elegir?

Entrevistas: Cuando necesitas información detallada de expertos

Talleres: Cuando hay muchos stakeholders con necesidades diversas

Prototipos: Cuando el sistema es innovador y no hay referencias

Encuestas: Cuando necesitas validar ideas con muchos usuarios

Tipos de Requisitos

Requisitos Funcionales

¿Qué debe hacer el sistema?

Describen las funciones y servicios que el sistema debe proporcionar.

Ejemplos:

• El sistema debe permitir login con email/contraseña

• Debe generar reportes en PDF

• Debe enviar notificaciones por email

• Debe calcular el precio con impuestos

Requisitos No Funcionales

¿Cómo debe funcionar el sistema?

Describen cualidades, restricciones y atributos de calidad.

Ejemplos:

• Tiempo de respuesta < 2 segundos

• Disponibilidad 99.9%

• Soportar 10,000 usuarios concurrentes

• Compatible con Chrome, Firefox, Safari

Rendimiento

Velocidad, throughput, latencia

Seguridad

Autenticación, autorización, encriptación

Escalabilidad

Capacidad de crecimiento

Usabilidad

Facilidad de uso, UX

Confiabilidad

Disponibilidad, tolerancia a fallos

Mantenibilidad

Facilidad de actualización

Importante

Los requisitos no funcionales son tan importantes como los funcionales. Un sistema que hace todo lo que debe pero es lento, inseguro o difícil de usar, es un sistema fallido.

Documentación de Análisis

Documento SRS (Software Requirements Specification)

El SRS es el entregable principal de la etapa de análisis. Especifica completamente qué debe hacer el software sin describir cómo lo hará.

Estructura Típica del SRS

  • 1. Introducción
    • Propósito y alcance
    • Definiciones y acrónimos
    • Referencias
  • 2. Descripción General
    • Perspectiva del producto
    • Funciones del producto
    • Características de usuarios
  • 3. Requisitos Específicos
    • Requisitos funcionales
    • Requisitos no funcionales
    • Restricciones
  • 4. Apéndices
    • Modelos y diagramas
    • Glosario

Características del SRS

  • Completo: No falta información
  • Correcto: Todos los requisitos son válidos
  • Consistente: Sin contradicciones
  • Inequívoco: Una sola interpretación
  • Verificable: Se puede comprobar
  • Modificable: Fácil de cambiar
  • Trazable: Origen y destino claros
  • Priorizado: Requisitos ordenados

Ejemplo de Requisito Bien Especificado:

REQ-001: El sistema debe validar que la contraseña del usuario

contenga al menos 8 caracteres, incluyendo al menos una letra

mayúscula, una minúscula, un número y un carácter especial.

Si no cumple, debe mostrar un mensaje de error específico

en menos de 0.5 segundos.

Consejo

La herramienta no hace al analista. Domina los conceptos primero, luego aprende las herramientas. Un buen analista puede trabajar incluso con papel y lápiz.

Mejores Prácticas

DO - Hacer

  • ✓ Involucrar a stakeholders desde el inicio
  • ✓ Documentar decisiones y suposiciones
  • ✓ Usar múltiples técnicas de elicitación
  • ✓ Validar requisitos frecuentemente
  • ✓ Priorizar requisitos con el cliente
  • ✓ Mantener trazabilidad completa
  • ✓ Crear prototipos cuando sea necesario
  • ✓ Revisar requisitos en equipo

DON'T - Evitar

  • ✗ Asumir que conoces todas las respuestas
  • ✗ Escribir requisitos ambiguos
  • ✗ Omitir requisitos no funcionales
  • ✗ Saltarse la validación con usuarios
  • ✗ Mezclar análisis con diseño prematuro
  • ✗ Ignorar las restricciones técnicas
  • ✗ Crear documentos que nadie lee
  • ✗ Trabajar en silos sin comunicación

Principios Fundamentales

1. Escucha más, habla menos: El análisis es 80% escuchar

2. Pregunta "¿Por qué?" 5 veces: Llega a la raíz del problema

3. Busca el problema, no la solución: No asumas soluciones

4. Documenta, documenta, documenta: La memoria es frágil

5. Valida temprano y frecuentemente: Falla rápido, aprende rápido

Recuerda

"El costo de corregir un error crece exponencialmente con cada fase del desarrollo. Invertir tiempo en el análisis ahorra dinero, tiempo y frustración más adelante."

Lección Aprendida

El 80% de los proyectos fallidos tienen un análisis deficiente como causa raíz. No es el código malo, son los requisitos equivocados.
